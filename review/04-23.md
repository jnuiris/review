# 4.23 面试题：多线程 · Task · async/await · 工业并发与超时

对应复习主题：`Thread` / `Task` / `async-await`；MES 多设备并发采集、异步日志、超时处理。

---

## 基础（5 道）

### 1. `Thread` 和 `Task` 有什么区别？什么场景还用 `Thread`？

**答：**  
- **`Thread`**：OS 线程，**重量级**，数量不宜多；适合**长时间阻塞**、需**线程专用**（如某些 COM STA）或**极底层控制**的场景。  
- **`Task`**：基于**线程池**的**工作单元**，适合**大量短任务**、**异步 IO** 配合 `async`。  
- **现代默认**：业务与 IO 优先 **`async/await` + Task**；只有明确需要**前台线程、固定线程模型**时才考虑 `new Thread`。  
- **MES**：成百上千台设备若每台一个 `Thread` 会浪费；多用 **线程池 + 异步 + 队列**。

---

### 2. `async`/`await` 在编译器层面大致做了什么？会不会新建线程？

**答：**  
- 方法被编译成**状态机**，在 `await` 处**挂起**，完成后**续执行**后续代码。  
- **不等于新开线程**：`await` 前后可能在**同一线程**（尤其 UI/ASP.NET 同步上下文），取决于 **Awaiter** 与 **SynchronizationContext**。  
- **`await Task.Run`** 才会把 CPU 工作丢到**线程池**。  
- **误区**：「async 就是多线程」——**错误**。

---

### 3. `Task.Run` 适合干什么？为什么不建议在 ASP.NET Core 里到处包一层？

**答：**  
- **适合**：把**纯 CPU 密集**计算从当前线程挪到**线程池**，避免阻塞调用线程。  
- **不适合**：用 `Task.Run` 包一层去「假装异步」**同步 IO**，会**白白占用线程池**，降低吞吐。  
- **正确做法**：IO 用 **真正异步 API**（`ReadAsync` 等）。  
- **MES 服务**：数据库、HTTP、Socket 优先 **async 原生方法**。

---

### 4. `ConfigureAwait(false)` 是干什么的？在什么库代码里常用？

**答：**  
- `await` 完成后是否**回到原同步上下文**（如 UI 线程、ASP.NET 旧版上下文）。  
- **`ConfigureAwait(false)`**：不捕获上下文，**任意线程**继续，减少**死锁**风险、略减开销。  
- **库代码 / 通用组件**里常用；**UI 事件处理**里若还要更新界面，通常**不**用 `false`（或之后 `Dispatcher`）。  
- **.NET Core / ASP.NET Core** 默认无传统同步上下文，**Mvc 里常可不写**，但**类库**仍习惯写。

---

### 5. 如何实现「异步超时」？请说出一种常用写法。

**答：**  
- **`CancellationTokenSource.CancelAfter(TimeSpan)`** + 将 **`CancellationToken`** 传入异步方法（如 `HttpClient.SendAsync`、自定义循环）。  
- 或 **`Task.WhenAny(task, Task.Delay(timeout))`**，再判断是否超时（注意**取消未完成任务**避免泄漏）。  
- **工业**：读 PLC、MQTT 请求应答，必须设**超时**，避免**永久挂死**。

## 进阶（5 道）

### J1. `async void` 除了 UI 事件，还有哪些坑？服务端为何禁止默认 `async void`？

**答：**  
- **异常**：无法被 `await`，易导致**未观察异常**。  
- **服务端**：API 应用 `async Task`/`ValueTask`，便于**中间件捕获**、**统一返回**。  
- **仅 UI** 等框架约定处用 `async void`，且建议**内部 try-catch**。

---

### J2. 多设备并发采集时，用「每设备一个 `Task`」和「单线程 + 轮询队列」各适合什么？

**答（概要）：**  
- **每设备一个长期运行的 `Task`**：适合 **独立长连接、真异步 IO**（Modbus TCP、MQTT、OPC UA 订阅），按「一台设备一个循环」好写；要控制**并发设备数**和**线程池/句柄**上限。  
- **单线程或少量工作线程 + 任务队列**：适合 **共享物理链路**（如 **RS-485 总线** 必须串行）、**设备极多**、或 **同步阻塞 API**（只能单线程顺序调），用队列排队，避免「千台设备千线程」。

---

**细说一：「每设备一个 `Task`」什么时候舒服、什么时候踩雷**

| 维度 | 说明 |
|------|------|
| **典型协议** | **Modbus TCP**（每 IP 一条 TCP）、**MQTT**（常共享 Broker 连接，业务上仍可按设备/主题分消费逻辑）、**OPC UA**（每服务器 Session + 订阅）。 |
| **优点** | 模型直观：**连接 → 循环读/订阅 → 解析 → 写内存/推 UI**；异常可**按设备隔离**，一台重连不拖垮整体结构。 |
| **风险** | 设备数从几十涨到几千时：**句柄、本机 ephemeral 端口、线程池调度**都会上来；若内部误用 **阻塞 `Sleep` 或同步读**，会占满线程池，表现为**全局卡顿**。 |
| **实践要点** | 用 **`async`/`await` + `CancellationToken`** 做周期与取消；全局限流（例如最多 **N** 台同时重连）；重连用 **`SemaphoreSlim`** 或 **Channel** 防抖，避免**惊群**。 |

---

**细说二：「单线程 + 队列 / 轮询」什么时候必选、怎么工作**

| 维度 | 说明 |
|------|------|
| **物理约束** | **RS-485 / 串口** 同一总线**同一时刻只能一问一答**；多从站必须 **严格串行** 轮询，天然适合 **一个 Writer + 队列**，按 round-robin 或优先级取「下一条读请求」。 |
| **设备极多 + 轻量轮询** | 上千台但**每秒只点检一次**、每次读很少：用 **少量定时器** 驱动「下一批从站」，比每台一个 `Task` 更省调度；本质是 **合并时间片**。 |
| **同步 DLL** | 厂商 **同步阻塞** API 若多线程同时调会**非线程安全**或**驱动崩**，只能 **单线程串行** 调，其它线程把请求 **入队**。 |
| **实现要点** | **`ConcurrentQueue`** / **`Channel`** 做待读队列；**单消费者** 出队 → 调协议 → 写结果；可 **专用线程** 或 **单 `Task` 死循环**，少和业务线程池抢。 |

---

**细说三：面试常追问——背压、限流、批量读**

- **背压（Backpressure）**  
  采集速度 **>** 下游（业务、DB、网络）处理能力时要有策略：**队列设上限**（满了丢最旧/拒绝新请求/降采样），避免**内存无限涨、OOM**。例：MQTT 洪峰时只保留**最新一条**设备状态用于界面。  

- **限流（Rate limiting）**  
  **单设备**：每秒最多几次轮询；**全局**：同时 TCP 连接数、重试频率上限。防止**重连风暴**或全厂扫描把 **PLC/串口服务器** 打挂。  

- **批量读（Batch）**  
  Modbus 一次读 **连续寄存器** 比多次零散读**报文更少、总线占用更低**；能合并的尽量 **一条请求多读**，485 上尤其明显。  

---

**面试一句话（可选用）**  
「**独立链路用异步 Task 好写；共享总线或阻塞 API 用单写者队列保序。** 规模上去靠 **限流、背压、批量读** 保设备端和本机都不爆。」

---

### J3. `lock` 和 `async` 能一起用吗？若要在异步代码里保护共享状态怎么办？

**答：**  
- **`lock` 内不能 `await`**（同一方法里持锁 await 会**死锁**或破坏锁语义）。  
- **做法**：缩小锁范围，只保护**同步内存**；或 **`SemaphoreSlim`** 的 **WaitAsync** 做**异步友好**的互斥；或 **Channel** 单写者。  
- **MES**：设备状态字典更新用 **`ConcurrentDictionary`** 或 **串行化到单队列** 处理。

---

### J4. 异步日志（如写入文件/队列）如何避免拖慢主业务？

**答：**  
- **内存队列 + 后台专用 Task** 批量刷盘；主路径只 **Enqueue**。  
- 使用成熟库（如 **Serilog Async sink**）或 **高性能队列**。  
- **注意**：进程崩溃可能丢**未刷盘**日志，关键告警可**同步写**或 **双写**。  
- **工业**：故障复盘依赖日志，**丢日志**与**性能**要权衡。

---

### J5. `Parallel.For` / `PLINQ` 与 `Task` 并行适用场景有何不同？

**答：**  
- **`Parallel`/`PLINQ`**：**数据并行**，适合** CPU 密集**、对集合分块处理；会占用**线程池**，默认**不适合 IO 阻塞**。  
- **`Task.WhenAll`**：多个**独立异步 IO** 任务并发。  
- **误用**：对 **IO 阻塞** 用 `Parallel.For` 会**耗尽线程池**，MES 服务整体**卡顿**。
