# 4.21 面试题：.NET 平台演进 · 生命周期 · GC 与分代 · LOH

对应复习主题：.NET Framework / .NET（Core）区别、跨平台、应用生命周期；垃圾回收、代回收、大对象堆（LOH）。

---

## 基础（5 道）

### 1. .NET Framework 和 .NET（.NET Core 及 5+）主要区别是什么？

**答：**  
- **Framework**：主要面向 **Windows**，与 Windows 生态深度绑定；版本随系统/安装包更新，**侧向兼容**包袱重。  
- **.NET Core / 现代 .NET**：**开源、跨平台**（Windows/Linux/macOS），**统一演进**（5/6/8/9），性能与部署（单文件、裁剪）更好，**长期支持（LTS）**节奏清晰。  
- **MES**：新系统多用 **.NET 6/8 WebAPI + 容器**；老产线客户端可能仍是 **Framework + WPF**，面试要能说明 **维护与迁移** 场景。

---

### 2. 「跨平台」在 .NET 里具体指什么？运行时扮演什么角色？

**答：**  
- **跨平台**：同一份 **IL（中间语言）** 由不同 OS 上的 **CoreCLR / runtime** 执行，配合 **BCL** 抽象文件、网络等差异。  
- **运行时**：负责 **JIT 编译**、**GC**、**类型系统**、**互操作** 等。部署时可带运行时（框架依赖）或 **自包含** 发布。  
- **工业**：Linux 上跑采集服务、边缘网关很常见，**跨平台**降低部署成本。

---

### 3. 简述 ASP.NET Core 应用启动到处理请求的「生命周期」里你关心的几个阶段。

**答：**  
- **Host 构建**：配置日志、配置源、依赖注入容器。  
- **ConfigureServices**：注册 **DbContext、HttpClient、后台服务** 等。  
- **Configure**：管道中间件顺序（认证、路由、异常）。  
- **请求**：中间件链 → Controller/Minimal API → 返回；**Scoped** 随请求创建释放。  
- **MES**：可在启动时注册 **与 PLC/MQTT 的长连接单例**，注意 **线程安全** 与 **优雅关闭**（`IHostApplicationLifetime`）。

---

### 4. 什么是托管堆？C# 为什么叫「托管」语言？

**答：**  
- **托管堆**：由 **CLR GC** 负责分配与回收的对象所在区域，开发者一般 **不手动 free**。  
- **托管**：内存生命周期由 **运行时管理**，配合 **类型安全** 与 **边界检查**，降低野指针问题（仍可能泄漏逻辑上的引用持有）。  
- **非托管**：`IntPtr`、P/Invoke、COM 等需 **显式释放** 或 **SafeHandle** 模式。

---

### 5. 分代 GC（Gen 0 / 1 / 2）和大对象堆（LOH）各是什么？为什么需要分代？

**答：**  
- **假设**：多数对象**短命**（临时对象），少数**长寿**。  
- **Gen 0**：新对象，回收频繁、速度快。  
- **Gen 1**：Gen 0 存活者晋升，缓冲带。  
- **Gen 2**：长期存活，回收**较慢**；与 **LOH** 常一起 Full GC。  
- **LOH**：**≥85000 字节**（当前实现）的大数组/大对象走 **LOH**，**默认不压缩**（历史行为，新版本有改进选项），易造成**碎片**。  
- **MES**：大批量 **DataTable/byte[]**、日志缓冲若过大，易压 **LOH** 与 **Gen2**，表现为**间歇性卡顿**。

## 进阶（5 道）

### J1. Server GC 与 Workstation GC 区别是什么？服务端默认用哪种？

**答：**  
- **Server GC**：为 **高吞吐、多核** 优化，**每逻辑处理器可有独立堆**，Full GC 时可能**暂停更长**但**总吞吐**更好；ASP.NET Core **默认 Server GC**（典型配置）。  
- **Workstation GC**：更关注**交互式延迟**，适合桌面/UI。  
- **上位机 WPF**：多为 **Workstation**；**MES 后台服务**：多为 **Server**。可通过 `runtimeconfig` 或 `gcServer` 调整（需验证场景）。

---

### J2. LOH 分配过多会导致什么问题？工业系统里如何缓解？

**答：**  
- **问题**：大对象**回收成本高**、易**碎片**、易触发 **Gen2/Full GC**，造成**延迟尖刺**。  
- **缓解**：避免频繁分配**巨大** `byte[]`/`string`；用 **ArrayPool** 租还缓冲；**分块**流式处理；缓存池化；监控 **LOH 分配率**。  
- **场景**：报文拼接、大批量导出 Excel、图像缓冲，都需警惕。

---

### J3. `IDisposable`、终结器（Finalizer）和 `using` 的关系？何时必须手写 `Dispose`？

**答：**  
- **`using`/ `await using`**：确定性释放 **非托管资源**（句柄、流、连接）。  
- **终结器**：**不确定**时机，GC 回收前调用，**成本高**，应作为**安全网**，正常路径用 `Dispose`。  
- **必须实现模式**：持有 **非托管句柄**、**需要及时释放** 的文件/套接字/数据库连接包装。  
- **工业**：设备 **串口/TCP 客户端** 包装类应 **`IDisposable`**，避免句柄泄漏。

---

### J4. 「析构函数 ~Foo()」和 `Dispose(false)` 在模式里怎么分工？

**答：**  
- C# **析构函数**生成 **Finalize**，由 **GC** 在对象不可达后**异步**调用。  
- 标准模式：`Dispose(bool disposing)` 中 **`disposing==true`** 时释放托管+非托管；**`false`** 时仅非托管（终结器路径不访问其他托管对象）。  
- **面试点**：避免在终结器里**阻塞**或**引用其他已终结对象**。

---

### J5. .NET 版本（如 net8.0）选 LTS 还是 STS？对 MES 项目选型有何建议？

**答：**  
- **LTS**：长期支持，适合**生产、长期维护**（如 8、6）。  
- **STS**：短期支持，新特性多，适合**尝鲜或短周期项目**。  
- **MES**：产线系统常选 **LTS + 明确升级窗口**；边缘服务可与公司统一 **容器基镜像** 对齐。
